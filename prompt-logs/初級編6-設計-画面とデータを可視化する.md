## 1.（8.1）対象ブラウザ要件の書き方 — ベストプラクティス

要件定義で対象ブラウザを書くときの定石は次の通りです。

**① バージョンは「固定番号」ではなく「相対表現」で書く**
Chrome/Edge は数週間ごとに更新されるため、「Chrome 120」のように固定で書くとすぐ陳腐化します。**「最新版および1つ前のメジャーバージョン（latest / latest-1）」** という相対表現が実務の標準です。

**② 対象は利用実態から決める**
個人利用なら本来「自分が使うブラウザ」だけで十分です。ただし今回は**インターネット公開**するので、将来自分が別端末（Mac/スマホ）から触る可能性も踏まえ、主要モダンブラウザ4種を押さえておくのが無難です。

**③ Internet Explorer は非対象と明記する**
IE は Microsoft がサポート終了済みです。「非対象」と一言書いておくと、実装時に無駄な互換対応をしなくて済みます。

**④ 要件を実装設定（Browserslist）に直結させる**
フロントエンド開発には `browserslist` という業界標準の対象ブラウザ指定の仕組みがあり、ビルドツールがこの設定に合わせてコードを自動調整します。要件定義に書いた方針（例：`last 2 versions, not dead`）をそのまま設定に落とせるので、**「ドキュメントの記述」と「実装の挙動」がズレない**のが利点です。

**私の推奨記述案:**
| 項目 | 内容 |
|------|------|
| 対象ブラウザ | Google Chrome / Microsoft Edge / Mozilla Firefox / Apple Safari |
| 対象バージョン | 各ブラウザの最新版および1つ前のメジャーバージョン（latest / latest-1） |
| 非対象 | Internet Explorer（サポート終了済み） |
| 実装への反映 | 上記方針を `browserslist` 設定に反映 |

## 2.（8.2）認証方式の比較 — ベーシック認証 vs アプリ本体への実装

前提（単一ユーザー・最低限・インターネット公開）を踏まえて比較します。

| 観点 | **ベーシック認証** | **アプリ本体への認証実装** |
|------|------------------|--------------------------|
| 仕組み | Webサーバー/ホスティング層でID・PWを要求（ブラウザ標準のダイアログ） | アプリ内にログイン画面を作り、セッション（Cookie等）で管理 |
| 実装コスト | **非常に低い**（アプリのコードをほぼ触らない） | 高い（ログイン画面・セッション・PWハッシュ保存・CSRF対策…） |
| 自作バグのリスク | 低い（枯れた仕組み） | **高い**（認証は脆弱性の急所。初学者には難所） |
| UI/UX | 素っ気ない（デザイン不可） | **自由にデザイン可能** |
| ログアウト | 作りにくい（ブラウザがキャッシュする） | 実装できる |
| 通信の安全性 | Base64のみ＝暗号化ではない。**HTTPS必須** | 同じくHTTPS前提だが実装で制御可能 |
| データモデルへの影響 | なし（user テーブル不要のまま） | パスワード等の保存が必要（DB or 環境変数） |
| 将来の複数ユーザー化 | 発展させにくい | **発展させやすい** |
| 今回の要件との相性 | ◎ 過不足なし・最速 | △ オーバースペック（学習/UI目的なら○） |

**補足（第3の選択肢）:**
技術スタック案（9章）で **BaaS（Supabase/Firebase）** が候補に挙がっています。BaaS の**組み込み認証機能**を使えば、自前実装の脆弱性リスクを負わずに、ログイン画面付き・将来の複数ユーザー対応も容易な認証を導入できます。「手軽さ」と「拡張性」の中間解です。

**私の推奨:**
「自分だけアクセスできればよい／最低限でよい」という要件に対しては、**ベーシック認証（またはホスティングのパスワード保護）が最もコスパが良い**です。実装が最速で、要件に対して過不足がありません（HTTPS必須の点だけ注意）。ただしバックエンドに BaaS を選ぶ場合は、**BaaS認証**の方が追加コストほぼゼロで拡張性も得られるため有力です。

## PCのマウス操作（現状イメージできているもの）

PCでは、カードを移動するとき次のようにします。

1. マウスでカードを**クリックしたまま**（ボタンを押しっぱなしに）する
2. そのままカーソルを目的の列（例：「作業中」）まで動かす
3. マウスのボタンを**離す** → その列にカードが移動し、ステータスが変わる

## スマホ・タブレットのタッチ操作（マウスがない代わり）

スマホ・タブレットにはマウスがありません。その代わりに、**画面を触る指**で同じことをします。

1. 動かしたいカードを**指で長押しする（ロングプレス、1秒ほど押し続ける）** → 「このカードをつかんだ」状態になり、カードが少し浮き上がる・影がつくなどの反応が出る
2. **指を画面につけたまま**、カードを目的の列までなぞって運ぶ
3. 移動先で**指を離す** → その列にカードが置かれ、ステータスが変わる

> イメージとしては、**スマホのホーム画面でアプリのアイコンを長押しして、別の場所へ移動させる、あの操作とほぼ同じ**です。

## なぜ「長押し」から始めるのか（スマホ特有の事情）

スマホでは、画面を指でなぞる操作は通常**「画面のスクロール」**に使われます。もし「触ってすぐドラッグ扱い」にすると、**画面を下にスクロールしたいだけなのにカードが動いてしまう**という誤操作が起きます。

そこで「**一定時間長押ししたらカード移動を開始**」というルールにすることで、
- 軽くスッとなぞる → 画面スクロール
- 長押ししてから動かす → カード移動

というふうに、指の同じ「なぞる」動きを2つの意味に使い分けられるようにするのが一般的です。これがスマホでのドラッグ＆ドロップの実態です。
