## 5. 機能要件

各機能について、概要・詳細仕様・受け入れ条件（実装完了の判断基準）をまとめます。

### 5.1 ボード管理

**概要**：タスクを分類する入れ物である「ボード」を作成・管理する機能。

**詳細仕様**

- ボードは複数作成できる（例：「仕事」「家事」など、用途ごとに分ける）。
- ボードには名前を設定できる。
- 画面上部のセレクトボックスから、表示するボード（または横断ビュー）を切り替えられる。初期選択は横断ビューとする（[6.2 ①](./03-screens.md#62-画面レイアウトワイヤーフレーム)参照）。
- ボードの作成・名称変更・削除・並べ替えは、セレクトボックス脇の管理ボタンから開く「ボード管理」画面で行う（削除時は所属するカードも削除される旨を確認する。[6.2 ②](./03-screens.md#62-画面レイアウトワイヤーフレーム)参照）。

**受け入れ条件**

- [ ] ボードを新規作成すると、セレクトボックスの選択肢に追加され、中身が空の状態（3列とも0件）で表示される
- [ ] ボード名を変更すると、セレクトボックス・詳細画面・横断ビューすべてに反映される
- [ ] ボードを削除すると、確認ダイアログが表示され、承認後に所属カードごと削除される
- [ ] 複数ボードが存在する状態で、セレクトボックスから任意のボード（または横断ビュー）へ切り替えられる

### 5.2 カード管理

**概要**：タスクの実体である「カード」を作成・編集・削除する機能。

**詳細仕様**

- カードは以下の属性を持つ。
  - タイトル（必須）
  - 説明・メモ（任意、自由記述）
  - 期日（任意）
  - ラベル（任意、複数付与可）
  - ステータス（未着手／作業中／完了、初期値は「未着手」）
- カードはいずれか1つのボードに所属する。
- カードの作成・編集・削除ができる。

**受け入れ条件**

- [ ] タイトルのみでカードを新規作成できる（他の属性は空でも作成可能）
- [ ] 作成したカードは、所属ボードの「未着手」列に表示される
- [ ] カード詳細を開き、説明・期日・ラベルを追加/変更できる
- [ ] カードを削除すると、一覧・横断ビュー・検索結果のすべてから消える

### 5.3 ドラッグ＆ドロップによるステータス変更

**概要**：カードをドラッグ＆ドロップ操作で「未着手 → 作業中 → 完了」の間で移動させる機能。スマートフォン・タブレットでは、これに加えて**ボタンやメニューによる明示的な操作手段**も提供する。

**詳細仕様**

- ボード詳細画面（3列のカンバン表示）で、カードをドラッグして別の列にドロップすると、そのカードのステータスが変わる。
- 列の順序は「未着手・作業中・完了」の固定順とし、順方向・逆方向どちらへの移動も可能とする（例：完了から作業中へ戻すこともできる）。
- 同一ステータス内での並び順（作成順、またはドラッグによる任意順）を保持する。
- マウスによるドラッグ操作に加え、スマートフォン・タブレットでの**タッチ操作によるドラッグ＆ドロップにも対応**する（[8.1](#81-対象デバイス画面サイズ)参照）。
- **スマートフォン・タブレット表示では、ドラッグ＆ドロップに加え、カードから移動先ステータス（未着手／作業中／完了）を選べる明示的な操作手段（例：カードの「移動」ボタンや「…」メニューから選択）を提供する。** 長押しによるドラッグは操作方法が視覚的に分かりづらいこと、カード数が多い場合はドラッグ距離が長くなり操作しづらいことへの対策とする。具体的なUIの形状（ボタン／メニュー等）は設計フェーズで決定する。

**受け入れ条件**

- [ ] 「未着手」列のカードを「作業中」列へドラッグすると、ステータスが即座に切り替わり、画面表示に反映される
- [ ] 「完了」列から「作業中」列へ戻す操作もできる
- [ ] ドラッグ操作後、ページを再読み込みしても移動後のステータスが保持されている（DBに保存されている）
- [ ] スマートフォン・タブレットにおいて、タッチ操作でカードをドラッグし、別の列へ移動できる
- [ ] スマートフォン表示において、ドラッグ操作を使わずにボタン／メニュー操作だけでカードのステータスを変更できる

### 5.4 横断マージビュー

**概要**：全ボードのカードを1つの画面に集約し、「未着手／作業中／完了」の3列の中をボードごとにグルーピングして表示する、本アプリの目玉機能。

**詳細仕様**

- 横断ビューは、ボード詳細画面（[5.3](#53-ドラッグドロップによるステータス変更)）と同じ「未着手／作業中／完了」の3列カンバンの枠組みを保ったまま、**各列の中身をボード単位でグループ化**して一覧表示する（[6.2 ③](./03-screens.md#62-画面レイアウトワイヤーフレーム)参照）。
- 各列内のボードごとのセクションには、そのボードに属するカード（タイトル・期日など）を表示する。
- 画面上部のセレクトボックスで「横断ビュー」を選択した状態がこの画面にあたる。任意のボードを選択すれば、そのボードのみの3列カンバン（[5.3](#53-ドラッグドロップによるステータス変更)）に切り替わる（[5.1](#51-ボード管理)参照）。
- 横断ビューからもカード詳細を開いて編集できる（元のボードを意識せず操作できる）。

**受け入れ条件**

- [ ] 横断ビューを開くと、「未着手／作業中／完了」の3列それぞれの中に、全ボードのカードがボードごとにグループ分けされて表示される
- [ ] あるボードでカードを新規作成すると、横断ビューの該当列・該当ボードのセクションにも反映される
- [ ] 横断ビュー上でカードを編集・ステータス変更すると、元のボード詳細画面にも反映される

### 5.5 ラベル管理

**概要**：カードを分類するための色付きラベルを作成・付与する機能。

**詳細仕様**

- ラベルは、あらかじめ用意された色パレットから色を選び、任意の名前（例：「優先度高」「買い物」）を付けて作成する（Trelloと同様の方式）。
- ラベルはボード単位で管理し、1枚のカードに複数のラベルを付けられる。
- カード一覧・横断ビューの両方で、付与されたラベルが色付きで表示される。

**受け入れ条件**

- [ ] 用意された色の中から選んでラベルを新規作成できる
- [ ] カードに複数のラベルを付与・解除できる
- [ ] ラベルを付けたカードは、一覧上でラベルの色が視認できる

### 5.6 期日の強調表示

**概要**：カードに設定した期日をもとに、期限切れ・期限間近を画面上で視覚的に警告する機能。

**詳細仕様**

- 期日を過ぎているカードは「期限切れ」として強調表示する（例：赤系の色）。
- 期日が近づいている（例：前日〜当日）カードは「期限間近」として強調表示する（例：黄系の色）。
- 強調は画面上の表示のみとし、ブラウザ通知やメール通知は行わない（[3.2](./01-overview.md#32-スコープ外の機能今回は実装しない)参照）。

**受け入れ条件**

- [ ] 期日を過ぎたカードが、通常のカードと視覚的に区別できる色で表示される
- [ ] 期日が当日・前日のカードが、期限切れとは別の色で強調表示される
- [ ] 期日未設定のカードには強調表示が付かない

### 5.7 アーカイブ

**概要**：完了したカードを削除せずに退避し、ボード表示をすっきり保ちながら後から見返せるようにする機能。

**詳細仕様**

- カードを「アーカイブする」操作を行うと、通常のボード表示・横断ビューから非表示になる。
- アーカイブされたカードは、専用の一覧画面からいつでも確認できる。
- アーカイブ一覧から、カードを元のステータスへ「復元」できる。
- アーカイブからの完全削除もできる。

**受け入れ条件**

- [ ] 「完了」列のカードをアーカイブすると、ボード表示・横断ビューから消える
- [ ] アーカイブ一覧を開くと、アーカイブ済みカードが確認できる
- [ ] アーカイブから「復元」すると、元のボードの該当ステータス列に戻る

### 5.8 検索・絞り込み

**概要**：キーワードやラベルを使って、目的のカードを素早く見つける機能。

**詳細仕様**

- カードのタイトル・説明に対するキーワード検索ができる。
- ラベルによる絞り込みができる。
- 検索・絞り込みは、単一ボード内・横断ビューの両方で利用できる。
- アーカイブ済みカードは、通常の検索結果には含めない（アーカイブ一覧内でのみ検索対象とする）。

**受け入れ条件**

- [ ] キーワードを入力すると、タイトルまたは説明に一致するカードのみが表示される
- [ ] ラベルを選択すると、そのラベルが付いたカードのみに絞り込める
- [ ] キーワードとラベルを組み合わせた絞り込みができる

## 8. 非機能要件

### 8.1 対象デバイス・画面サイズ

- PC・タブレット・スマートフォンでの利用を対象とし、**レスポンシブ対応（画面幅に応じたレイアウト最適化）をスコープに含める**。
- 具体的なブレークポイント（px値）は実装フェーズで確定するが、方針の目安は以下の通り。PC版・スマートフォン版それぞれのレイアウトイメージは[6.2 ②](./03-screens.md#62-画面レイアウトワイヤーフレーム)を参照。

| 区分           | 画面幅の目安  | レイアウト方針                                                 |
| -------------- | ------------- | ---------------------------------------------------------------- |
| PC             | 1280px 程度〜 | 「未着手/作業中/完了」の3列を横並びのカンバン表示              |
| タブレット     | 768px 程度〜  | 3列カンバンを維持、または縦積みに切り替え（実装時に検証）      |
| スマートフォン | 〜767px 程度  | 3列を縦積み表示に切り替え、1列ずつスクロールして確認・操作する |

- ドラッグ＆ドロップ操作は、マウス操作に加え**タッチ操作にも対応**する（[5.3](#53-ドラッグドロップによるステータス変更)参照）。
- ただし、スマートフォンでの長押しドラッグは操作方法が分かりづらく、カード数が多いと移動距離も長くなるため、**狭い画面ではドラッグ＆ドロップに加えて、移動先を選べるボタン等の明示的な操作手段を提供する**（[5.3](#53-ドラッグドロップによるステータス変更)参照）。

**対象ブラウザ・バージョン**

| 項目           | 内容                                                                                                     |
| -------------- | ---------------------------------------------------------------------------------------------------------- |
| 対象ブラウザ   | PC：Google Chrome / Microsoft Edge / Mozilla Firefox / Apple Safariモバイル：iOS Safari / Android Chrome |
| 対象バージョン | 各ブラウザの最新版および1つ前のメジャーバージョン（**latest / latest-1**）                         |
| 非対象         | Internet Explorer（サポート終了済みのため対象外とする）                                                  |

> **なぜバージョンを固定値で書かないのか？**
> Chrome・Edge 等は数週間おきに自動更新されるため、「Chrome 120」のように特定のバージョン番号で固定すると、要件定義書の記述がすぐに実態と合わなくなります。そのため「最新版とその1つ前（latest / latest-1）」という相対的な表現を使うのが一般的です。この方針はフロントエンドのビルドツールが参照する `browserslist` という設定（例：`last 2 versions, not dead`）にそのまま反映でき、**要件定義書の記述と実装の挙動を一致させられる**という利点があります。

### 8.2 認証・セキュリティ

- 本アプリは**インターネット上に公開**し、自分専用のツールとして利用する想定であるため、**本人以外がアクセスできないよう最低限の認証を設ける**。
- 認証方式は**ベーシック認証**を採用する。Webサーバー／ホスティング層（[9章](./05-tech-stack-and-roadmap.md#9-技術スタック案)参照）でID・パスワードを要求する方式とし、アプリ本体の実装（画面・データモデル）には手を加えない。
- ベーシック認証の採用にあたり、以下を必須とする。
  - **HTTPS（TLS）による通信の暗号化を必須とする**：ベーシック認証はID・パスワードをBase64でエンコードして送るだけであり、それ自体には暗号化の効果がないため、HTTPS化を前提とする。
  - **認証情報（ID・パスワード）はコードに直書きせず、環境変数等で管理する**。
- 単一ユーザー利用を前提とした簡易的な認証であり、ログイン画面・ログアウト・パスワード再発行といった機能は持たない。
- 将来、複数ユーザーに対応する、またはログイン画面などのUIを用意したくなった場合は、アプリ本体への認証機能の実装へ移行する（[10章](./05-tech-stack-and-roadmap.md#10-今後の拡張ロードマップ)参照）。その場合も、データモデルは `user_id` を後から追加できる形を維持しているため（[7.3](./04-data-model.md#73-設計上の補足)参照）、移行はスムーズに行える想定である。

### 8.3 性能

- カードが数百件程度に増えても、一覧表示・検索・ドラッグ＆ドロップ操作が体感的にストレスなく行えること（主要な操作の反映が概ね1秒以内）を目安とする。

### 8.4 データ永続化

- 入力したデータはブラウザを閉じたり再読み込みしても失われないよう、サーバー側のデータベースに保存する（[9章](./05-tech-stack-and-roadmap.md#9-技術スタック案)で候補を検討）。

### 8.5 保守性・拡張性

- 将来的にアプリ本体への認証機能（複数ユーザー対応）やメンバー機能を追加しやすいよう、データモデルは `user_id` 等のカラムを後から追加できる形を意識して設計する（[7.3](./04-data-model.md#73-設計上の補足)参照）。
- 画面・API は機能単位（ボード／カード／ラベル／アーカイブ／検索）で分離し、機能追加時の影響範囲を小さく保つ。
